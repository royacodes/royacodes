# متد HTTP QUERY رسماً استاندارد شد؛ چرا این اتفاق برای طراحی APIها مهم است؟

*بر اساس انتشار RFC 10008 توسط IETF در ژوئن ۲۰۲۶*

## مقدمه

در ژوئن ۲۰۲۶، **IETF** با انتشار **RFC 10008**، متد جدید **HTTP QUERY** را به‌عنوان بخشی از استاندارد HTTP معرفی کرد.

در نگاه اول ممکن است این تغییر کوچک به نظر برسد، اما در واقع یکی از خلأهای قدیمی طراحی APIها را برطرف می‌کند: **اجرای عملیات Query بدون تغییر داده‌ها، همراه با امکان ارسال Request Body.**

این متد قرار نیست شیوه توسعه APIها را متحول کند؛ بلکه هدف آن شفاف‌تر کردن مفهوم درخواست‌ها و نزدیک‌تر شدن به معنا و فلسفه اصلی HTTP است.

---

## مشکل GET و POST چه بود؟

سال‌هاست برای دریافت اطلاعات از دو متد استفاده می‌کنیم:

- **GET** برای دریافت داده
- **POST** برای جستجوهای پیچیده

اگرچه GET از نظر معنایی بهترین گزینه برای خواندن اطلاعات است، اما محدودیت‌هایی دارد:

- محدودیت طول URL
- مناسب نبودن برای فیلترهای پیچیده
- نمایش پارامترها در URL
- دشواری ارسال ساختارهای بزرگ JSON

به همین دلیل بسیاری از APIها از الگوی زیر استفاده می‌کنند:

```http
POST /users/search
```

در حالی که این درخواست هیچ تغییری در داده‌ها ایجاد نمی‌کند، تنها به این دلیل از POST استفاده می‌شود که امکان ارسال Request Body را فراهم می‌کند.

---

## متد QUERY چه تغییری ایجاد می‌کند؟

متد **QUERY** دقیقاً برای چنین سناریوهایی طراحی شده است.

این متد مانند GET یک عملیات **Safe** محسوب می‌شود؛ یعنی اجرای آن نباید باعث تغییر وضعیت سرور شود.

اما برخلاف GET، امکان ارسال Request Body را نیز فراهم می‌کند.

نمونه:

```http
QUERY /users
Content-Type: application/json

{
  "department": "Engineering",
  "experience": {
    "min": 5
  },
  "skills": ["Go", "Rust"]
}
```

در این حالت، هدف درخواست کاملاً مشخص است: اجرای یک پرس‌وجو، نه ایجاد یا ویرایش اطلاعات.

---

## مزایای QUERY

### شفافیت بیشتر

خود متد HTTP هدف درخواست را مشخص می‌کند.

- GET → دریافت منبع
- QUERY → اجرای پرس‌وجو
- POST → ایجاد یا پردازش داده

### طراحی تمیزتر API

دیگر نیازی نیست POST برای عملیات Search مورد استفاده قرار گیرد و API از نظر معنایی استانداردتر خواهد بود.

### مستندسازی بهتر

توسعه‌دهندگان و مصرف‌کنندگان API راحت‌تر متوجه می‌شوند کدام Endpoint فقط داده را می‌خواند و کدام Endpoint باعث تغییر اطلاعات می‌شود.

### آینده بهتر برای اکوسیستم HTTP

اکنون فریم‌ورک‌ها، API Gatewayها، ابزارهای توسعه و سایر اجزای زیرساخت می‌توانند به‌صورت اختصاصی از عملیات Query پشتیبانی کنند؛ بدون اینکه مجبور باشند آن را به‌عنوان یک درخواست POST تفسیر کنند.

---

## وضعیت فعلی

هرچند QUERY اکنون یک استاندارد رسمی HTTP است، اما پذیرش آن در اکوسیستم نرم‌افزار زمان‌بر خواهد بود.

قبل از استفاده در محیط Production بهتر است بررسی کنید که:

- Framework مورد استفاده
- HTTP Client
- API Gateway
- Reverse Proxy
- Load Balancer
- ابزارهای امنیتی

از این متد پشتیبانی می‌کنند.

احتمالاً تا مدتی بسیاری از APIها همچنان برای عملیات جستجوی پیچیده از POST استفاده خواهند کرد.

---

## جمع‌بندی

استاندارد شدن متد **HTTP QUERY** شاید بزرگ‌ترین تغییر HTTP در سال‌های اخیر نباشد، اما یکی از منطقی‌ترین آن‌هاست.

وجود متدی اختصاصی برای عملیات Query باعث می‌شود طراحی APIها خواناتر، مستندسازی آن‌ها ساده‌تر و معناشناسی HTTP دقیق‌تر شود.

گاهی اوقات، بزرگ‌ترین پیشرفت‌ها از شفاف‌تر کردن مفاهیم به دست می‌آیند.
